>_ EXECUTIVE SUMMARY

Sidekick LLM is a robust, privacy-first desktop application designed to bridge the gap between complex local AI models and user-friendly interfaces. By shifting inference from the cloud to the edge via Ollama, it guarantees absolute data sovereignty and offline availability.

Built as a unified desktop architecture using the Electron framework, Sidekick LLM demonstrates advanced full-stack engineering by securely sandboxing web views, establishing high-performance Inter-Process Communication (IPC) for token streaming, and implementing zero-latency localized search.

>_ ARCHITECTURE & SECURITY TOPOLOGY

  • Renderer Process (React 18 + Vite): Handles state transitions, UI rendering, and incremental streaming message updates. Strictly isolated from Node.js APIs.
  • Main Process (Node.js): Orchestrates local host integrations, managing direct HTTP interfacing with the Ollama daemon and executing file system operations.
  • IPC Security Bridge: A context-isolated preload script (preload.cjs) defines a narrow, strictly typed API, preventing Remote Code Execution (RCE) and XSS vulnerabilities from escaping the web view into the host OS.

>_ CORE ENGINEERING CHALLENGES

1. Reactive Token Streaming over IPC: Cloud LLMs stream tokens via Server-Sent Events (SSE) directly to the browser. In a desktop app, the main process must proxy the local LLM's HTTP chunked response. I engineered a streaming event pipeline over the Electron IPC bridge, piping generated tokens synchronously to React state to provide a low-latency, engaging UI without memory bloat or backpressure bottlenecks.

2. Synchronous Database Performance: For single-tenant desktop applications, asynchronous database queries (Promises) introduce unnecessary V8 event loop overhead. I utilized better-sqlite3, relying on its C++ bindings to execute SQLite operations synchronously. This simplifies repository logic and drastically reduces query latency for chat history retrieval.

3. High-Performance Local Search (FTS5): Managing massive local chat logs requires robust search capabilities. Instead of integrating a heavy local vector database, I utilized SQLite's FTS5 virtual tables. Triggers automatically index conversations and messages, providing zero-latency, full-text offline search capabilities directly within the SQL layer.

4. Resilient Offline Architecture: Because edge computing depends on local hardware, daemons like Ollama can crash or be terminated. The application employs intelligent polling mechanisms and graceful degradation to monitor AI health, ensuring the user is immediately informed of daemon state changes without silent UI failures.
