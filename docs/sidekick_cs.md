# Case Study: Sidekick LLM

## Overview
Sidekick LLM is a robust, privacy-first desktop application designed to provide a seamless chat interface for local Large Language Models (LLMs) via Ollama. It combines the responsiveness of a modern web interface with the native capabilities of a desktop environment, enabling offline operation, persistent conversation history, and high-performance local search.

## Architectural Design
The application has been engineered using the **Electron** framework, moving away from a traditional client-server web model to a unified desktop architecture.

### 1. Frontend (Renderer Process)
- **Framework**: React 18 and Vite for rapid development and optimized builds.
- **State Management**: React Hooks (`useState`, `useEffect`) handle state transitions, streaming message updates, and Ollama connection status.
- **IPC Bridge**: The UI communicates securely with the Electron main process via a context-isolated preload script (`window.sidekick`).

### 2. Backend (Main Process)
- **Runtime**: Node.js within Electron.
- **Local Integration**: Directly interfaces with the local Ollama HTTP API (`http://localhost:11434`) for model inference without relying on external cloud services.
- **Inter-Process Communication (IPC)**: A structured IPC layer (`ipc.cjs`) handles requests from the renderer for database operations, model management, and file exports.

### 3. Data Persistence & Search
- **Database**: SQLite (via `better-sqlite3`) provides synchronous, high-performance local data storage. 
- **Schema**: Stores `conversations`, `messages`, and `settings`. Includes triggers for automatic indexing.
- **Full-Text Search (FTS5)**: Utilizes SQLite's FTS5 virtual tables to implement lightning-fast, offline-capable local search across conversation histories.

## Engineering Patterns

### 1. Secure Context Isolation
The application strictly enforces security best practices by separating the Node.js environment from the web environment. The renderer process has no direct access to Node APIs or the filesystem; it relies on a narrowly defined API exposed through `preload.cjs`.

### 2. Reactive Streaming
The integration with Ollama leverages a streaming API pattern. As tokens are generated by the local model, the main process pipes them through the IPC bridge via events, allowing the React frontend to incrementally update the UI. This provides a low-latency, engaging user experience similar to cloud-based LLM chats.

### 3. Synchronous Database Operations
By using `better-sqlite3`, the main process handles database queries synchronously. In a desktop context with a single active user, this avoids the overhead of asynchronous promises for data fetching, simplifying the repository logic while maintaining high performance.

### 4. Resilient Offline Architecture
Since the model (Ollama) and the database (SQLite) are both local, the application operates fully offline. It includes robust error handling and polling mechanisms to gracefully handle Ollama's availability, ensuring the user is informed if the model server drops or is unavailable.

## Conclusion
Sidekick LLM successfully bridges the gap between complex local AI models and user-friendly desktop interfaces. By strategically leveraging Electron, React, and SQLite, it delivers a secure, offline-first, and highly responsive AI assistant that prioritizes user privacy and data ownership.
